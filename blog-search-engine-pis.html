How I Built a Search Engine on Raspberry Pis

FTS5, D1, Workers AI. Full-text search with AI-generated answers, running on self-hosted infrastructure for effectively $0 marginal cost.

The Problem

BlackRoad OS has 629 repos, 200 agents, 18 domains, and 45 web pages. Finding anything meant grep or memory. I needed a search engine that could answer questions like "how does the WireGuard mesh work?" with actual context, not just keyword matches.

The options were Algolia ($250+/month at scale), Elasticsearch (heavy, needs dedicated servers), or Meilisearch (good but another service to manage). All of them add a dependency I don't control.

The Stack

  • FTS5 — SQLite's full-text search extension, built into D1 (Cloudflare's serverless SQLite)
  • D1 — stores the search index, zero maintenance, $0 up to 5GB
  • Workers AI — Llama 3.1 8B generates human-readable answers from search results
  • KV — caches popular queries for 1 hour

The entire search engine is a single Cloudflare Worker. No servers. No containers. Under 50KB of code.

How It Works

When you search, the Worker:

  • Checks KV cache first (cache hit = instant response)
  • Runs an FTS5 query against the D1 index: SELECT * FROM pages WHERE pages MATCH ? ORDER BY rank
  • Takes the top 5 results and constructs a context prompt
  • Sends context + question to Workers AI (Llama 3.1 8B) for answer generation
  • Returns both the AI answer and the raw search results
  • Caches the response in KV for 1 hour

Response time for a cache miss: 800ms-1.5s. Cache hit: under 50ms.

The Index

The search index contains 200+ pages crawled from all 18 domains. Each page is stored with its title, URL, content (stripped of HTML), and a summary. The indexer runs every 6 hours via cron on the Mac, rebuilding the FTS5 index from scratch.

For local search (the br search-all CLI), there's a separate Python FTS5 index with 1,383 entries from 23 different sources: codex, TILs, journal, websites, repos, tools, agents, wiki, scripts, and more.

Why FTS5 Over Everything Else

FTS5 is underrated. It handles boolean queries, phrase matching, prefix matching, and BM25 ranking — all in SQLite. No separate service. No cluster. No configuration.

Since D1 is just SQLite under the hood, FTS5 works natively. Creating the index is a single SQL statement:

CREATE VIRTUAL TABLE pages USING fts5(title, url, content, summary);

That's it. Insert your pages. Query with MATCH. SQLite handles tokenization, stemming, and ranking.

For vector search (semantic similarity), we use Qdrant on Alice (the gateway Pi) with nomic-embed-text embeddings. But for 95% of queries, FTS5 keyword matching is faster and more predictable than vector search.

What It Costs

Effectively nothing beyond the Cloudflare Workers plan ($5/month, shared with all other Workers):

  • D1: free up to 5GB (index is under 10MB)
  • Workers AI: included in the plan
  • KV: free tier covers caching
  • Worker execution: well within free tier limits

Compare that to Algolia at $1/1000 search requests, or running your own Elasticsearch cluster.
